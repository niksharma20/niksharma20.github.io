Blueprint: Quality of Service (QoS)

The Quality of Service (QoS) class is an implicit contract with Kubernetes about a workload's importance. It directly determines which pods are sacrificed to maintain node stability during periods of resource scarcity, such as a memory shortage.

QoS Classes and Eviction Priority

When a node is under pressure, the `kubelet` evicts pods to reclaim resources. The QoS class determines the order of eviction. Understanding this hierarchy is critical for protecting important workloads.

1. Guaranteed (Highest Priority)

Achieved when `requests` == `limits` for all resources in all containers. These pods are the last to be evicted.

2. Burstable (Medium Priority)

Achieved when `requests` < `limits`. These pods have a baseline guarantee but can burst to use more resources. Evicted after BestEffort.

3. BestEffort (Lowest Priority)

Assigned when no `requests` or `limits` are set. These pods have no guarantees and are the first to be evicted.

QoS as an Implicit Resiliency Contract

Setting resource requests and limits is a direct declaration of a workload's importance. The QoS class is how Kubernetes honors that declaration during resource contention.

The Developer's Declaration

By omitting requests and limits, a developer implicitly tells Kubernetes: "This workload is not critical and can be terminated first." Conversely, setting `requests` equal to `limits` declares: "This workload is mission-critical. Protect it from eviction at all costs." Failing to set these values is a common anti-pattern that results in production services being treated as expendable.
