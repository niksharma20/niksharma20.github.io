Blueprint: Automated Recovery

Kubernetes' ability to automatically recover from failures is one of its most powerful features. This is driven by the actions the `kubelet` takes in response to probe failures, from temporarily removing a pod from service to restarting a container that is completely broken.

Probe-Driven Recovery Actions

Each probe type triggers a distinct recovery action, allowing for nuanced control over the self-healing process.

Readiness Probe → Remove from Service

When a Readiness probe fails, the `kubelet` takes the pod's IP address out of the corresponding Service's endpoints. This is a non-destructive action that isolates the pod from traffic while it recovers, without killing it. This is the correct way to handle temporary issues or dependency failures.

Liveness Probe → Restart Container

When a Liveness probe fails, the `kubelet` assumes the application is in an unrecoverable state (e.g., deadlocked) and kills the container. The container is then restarted according to the pod's `restartPolicy`. This is a destructive, last-resort action.

Startup Probe → Delay Other Probes

The Startup probe's primary recovery action is preventative. By succeeding, it allows the other probes to begin. If it fails, it also triggers a container restart. Its purpose is to protect slow-starting applications from being killed by a premature liveness probe failure.
