Blueprint: North-South & East-West Traffic

Understanding and managing the flow of network traffic is central to both the security and reliability of a Kubernetes application, distinguishing between traffic entering the cluster and traffic moving within it.

Managing Traffic Flows

The new Gateway API is succeeding Ingress for North-South (external) traffic, offering a more powerful and portable standard. A Service Mesh is the standard for securing and controlling East-West (internal) traffic.

North-South (Ingress/Egress)

Managed by an API Gateway or Ingress Controller. The Gateway API provides a role-oriented model that is ideal for multi-team environments, separating infrastructure concerns from application routing.

User → Internet → [Gateway API] → Service

East-West (Internal)

Managed by a Service Mesh (e.g., Istio, Linkerd). The mesh provides mTLS security, observability, and fine-grained traffic control (retries, timeouts) for all service-to-service communication via sidecar proxies.

Service A → [Service Mesh Sidecar] → Service B

Convergence: The GAMMA Initiative

The GAMMA (Gateway API for Mesh Management and Administration) initiative aims to unify North-South and East-West traffic management by using the same Gateway API resources (e.g., `HTTPRoute`) to configure both the API Gateway and the Service Mesh. This promises a future with a single, portable API for all L7 routing.

A Unified Routing Model

In the GAMMA model, an `HTTPRoute` can be attached to a `Gateway` to manage ingress traffic, or attached directly to a `Service` to manage internal, mesh-controlled traffic. This eliminates API fragmentation and cognitive overhead for developers.
